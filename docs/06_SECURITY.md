# Operation PAWS: Security Architecture
**Classification: PUBLIC KNOWLEDGE — The security does not rely on this being secret.**

## Core Principle
> "Security through strength, not secrecy."

The source code, database schema, and this document are fully public. The system remains secure because its locks are **cryptographic and server-side**, not logical secrets hidden in JavaScript.

## Attack Vectors vs. Our Defenses

| Attack | Our Defense |
| :--- | :--- |
| Read the source code to find the PIN | There is no PIN in the source. Auth is via Supabase JWT tokens issued server-side. |
| Directly query the Supabase REST API | Row Level Security (RLS) rejects all requests without a valid session token. |
| Brute-force the login | Edge Function rate-limiting: 5 attempts per IP per 15 minutes, then hard block. |
| Replay a valid JWT token | Tokens expire in 60 minutes. Refresh tokens are single-use. |
| Delete an audit log to cover tracks | The `paws_audit_log` table has RLS: INSERT only. No UPDATE or DELETE, ever. |
| Social engineer a PIN out of an officer | Officers have role-based JWT claims. A field officer token cannot call commissioner-level functions. |

## Access Tier Architecture

```
PUBLIC INTERNET
      │
      ▼
[ GitHub Pages / HTML ] ──→ No sensitive data here. Static only.
      │
      ▼
[ Supabase Edge Functions ] ──→ Rate-limited. IP-blocked after 5 fails.
      │                          JWT validated on EVERY request.
      ▼
[ Supabase Auth Layer ] ──→ Issues short-lived JWTs with role claims.
      │                      Roles: public | officer | commissioner
      ▼
[ PostgreSQL + RLS ] ──→ Even if you have the DB connection string,
                           RLS rejects your query if your JWT role
                           does not match the policy.
```

## Role Definitions (JWT Claims)

| Role | Can Read | Can Write | Can Delete |
| :--- | :--- | :--- | :--- |
| `public` | Tracker, Verify | Nothing | Never |
| `officer` | Own intake records | New intake records | Never |
| `commissioner` | All records | Approve/Reject status | Only with 2FA TOTP |

## Two-Factor Authentication
- Commissioner-level destructive actions (DELETE) require a **TOTP code** generated by an authenticator app (Google Authenticator / Authy).
- The TOTP secret is stored **only in the Supabase Vault** (encrypted at rest). It is never exposed to the frontend.
- Verification happens inside an Edge Function. The client sends the code; the function validates it server-side and issues a short-lived (5-minute) `action_token`.

## Audit Log (Immutable)
Every action writes a row to `paws_audit_log`:
```sql
-- RLS Policy: Allow INSERT for authenticated users.
-- Allow SELECT for commissioner role.
-- DENY UPDATE and DELETE for everyone, including service_role.
```
This means the audit trail cannot be falsified.
